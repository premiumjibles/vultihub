# SDK: Add missing agent-required API methods

## Objective

Add API methods to @vultisig/sdk that the agent chat's 40 tool handlers require but the SDK does not yet expose. Without these, the MCP server and standalone chat app cannot replicate the full agent feature set.

## Context & Findings

The SDK (v0.5.0) already covers the majority of agent operations: signing, balances, swaps, gas estimation, token/chain management, address book, and security scanning. However, 6 categories of agent tool handlers have no SDK equivalent:

**Policy management** (5 tools: `policyList`, `policyStatus`, `policyAdd`, `policyDelete`, `policyGenerate`)
- These call the Verifier API at `https://verifier.vultisig.com`
- Implementation in vultisig-windows: `core/ui/agent/tools/handlers/policy*.ts`
- Shared helpers: `core/ui/agent/tools/shared/policyHelpers.ts`
- Reference: `core/ui/agent/tools/shared/verifierApi.ts` for HTTP client pattern

**Plugin management** (5 tools: `pluginList`, `pluginSpec`, `pluginInstalled`, `pluginInstall`, `pluginUninstall`)
- Also calls the Verifier API
- Implementation: `core/ui/agent/tools/handlers/plugin*.ts`
- Config: `core/ui/agent/tools/shared/pluginConfig.ts`

**EVM contract reads** (`readEvmContract`)
- Executes read-only `eth_call` against arbitrary EVM contracts
- Implementation: `core/ui/agent/tools/handlers/readEvmContract.ts`
- Uses viem for ABI encoding/decoding

**Thorchain queries** (`thorchainQuery`)
- Queries Thorchain node APIs (pools, inbound addresses, mimir, etc.)
- Implementation: `core/ui/agent/tools/handlers/thorchainQuery.ts`

**Transaction history** (`transactionHistory`)
- Chain-specific tx history fetching and parsing
- Implementation: `core/ui/agent/tools/handlers/transactionHistory.ts`

**Custom transaction building** (`buildCustomTx`)
- Build arbitrary EVM contract interaction keysign payloads
- Implementation: `core/ui/agent/tools/handlers/buildCustomTx.ts`
- Uses: `core/mpc/keysign/` payload builders

**Rejected approach:** Wrapping the raw handler code directly in the MCP — this would re-introduce monorepo coupling. Methods should be proper SDK citizens with consistent error handling and types.

## Acceptance Criteria

- [ ] `vault.getPolicies()`, `vault.getPolicyStatus(id)`, `vault.addPolicy(params)`, `vault.deletePolicy(id)`, `vault.generatePolicy(params)` methods exist and call verifier API
- [ ] `sdk.getPlugins()`, `sdk.getPluginSpec(id)`, `vault.getInstalledPlugins()`, `vault.installPlugin(id)`, `vault.uninstallPlugin(id)` methods exist
- [ ] `vault.readEvmContract(chain, contractAddress, abi, functionName, args)` returns decoded result
- [ ] `vault.queryThorchain(endpoint, params)` returns parsed JSON
- [ ] `vault.getTransactionHistory(chain, options?)` returns typed transaction list
- [ ] `vault.prepareCustomTx(chain, contractAddress, abi, functionName, args, value?)` builds a keysign payload
- [ ] All new methods have TypeScript types exported from `types/index.ts`
- [ ] Unit tests for each new service
- [ ] `yarn check:all` passes

## Gotchas

- Policy/plugin APIs hit the Verifier service, not the FastVault API — different base URL
- Transaction history parsing is chain-specific — needs resolvers for EVM, UTXO, Cosmos, Solana at minimum
- `readEvmContract` needs ABI parsing — viem's `decodeFunctionResult` is already a dependency
- Custom TX building reuses the existing `prepareSendTx` pipeline but with arbitrary calldata
