# SDK: Encrypted vault file loading with local password

## Objective

Add a vault file loading API to @vultisig/sdk that decrypts the local key share from an encrypted file on disk using a user-provided password. This is the first half of the two-password auth model needed for headless/CLI/MCP usage where no UI exists to manage vault state.

## User Story

A developer using the SDK programmatically (via MCP or CLI) needs to load their vault from a file, decrypt the local key share with a password, and have it ready for signing — without a desktop app running.

## Design Constraints

- The SDK already has storage abstractions: `FileStorage`, `BrowserStorage`, `MemoryStorage` in `packages/sdk/src/storage/`
- Vault encryption/decryption already exists — `VaultBase` has `lock()`, `unlock()`, `isUnlocked()`, `isEncrypted`
- The desktop app stores vaults via Wails Go backend (`vultisig-windows/storage/store.go`) using SQLite — the SDK needs to read exported vault files (`.vult` format), not the SQLite DB directly
- `importVault()` already exists on the `Vultisig` class — may be the right foundation
- Must work in Node.js (for MCP server) — no browser/Wails dependencies

## Context & Findings

**Current vault loading flow (desktop app):**
1. Go backend reads vault from SQLite → passes to frontend via Wails
2. Frontend decrypts key shares in memory using vault password
3. Key shares held in `walletContext` for signing session

**Current SDK vault loading:**
- `Vultisig.importVault(backup)` — imports from VaultBackup object
- `vault.unlock(password)` — decrypts an encrypted vault
- `FileStorage` — reads/writes vault JSON to disk
- Missing: a single `open(path, password)` convenience API

**What's needed for headless usage:**
1. A config file (`~/.vultisig/config.json`) pointing to vault file path and server URLs
2. A discovery mechanism to find existing vault files (from desktop app exports)
3. A `Vault.open(path, password)` or `sdk.loadVault(path, password)` convenience method
4. Password held in memory only — never written to disk by the SDK

**Rejected: reading SQLite directly** — the desktop app's Go SQLite storage is not portable. Vault export (`.vult` files) is the right interchange format.

## Files

SDK files to modify/create (in vultisig-sdk):
- `packages/sdk/src/Vultisig.ts` — add `loadVault(path, password)` method
- `packages/sdk/src/storage/FileStorage.ts` — reference for file I/O patterns
- `packages/sdk/src/vault/VaultBase.ts` — reference for `unlock()` flow
- `packages/sdk/src/config/` — new directory for config file handling
- `packages/sdk/src/types/index.ts` — config types

Reference (vultisig-windows):
- `vultisig-windows/core/ui/agent/tools/shared/walletContext.ts` — how vault metadata is held in memory
- `vultisig-windows/storage/store.go` — desktop vault storage (for understanding export format)

## Acceptance Criteria

- [ ] `sdk.loadVault(filePath, password)` loads a `.vult` file and returns an unlocked `FastVault` or `SecureVault`
- [ ] Throws typed `VaultError` with clear message if file not found, wrong password, or corrupt file
- [ ] Config file support: `sdk.loadConfig(configPath?)` reads `~/.vultisig/config.json` with `{ vaultPath, serverUrl?, relayUrl? }`
- [ ] `sdk.discoverVaults(searchPaths?)` scans common locations for `.vult` files and returns list with metadata (name, type, chains)
- [ ] Password is never persisted to disk — only held in the vault instance memory
- [ ] Works in Node.js (no browser APIs)
- [ ] Unit tests for load, config, and discovery
- [ ] `yarn check:all` passes

## Gotchas

- `.vult` export format may include both encrypted and unencrypted variants — check `VaultBackup` type
- The SDK's `unlock()` method may need a time-based auto-lock — for MCP usage, consider an option to keep unlocked for the session lifetime
- `discoverVaults()` should check `~/.vultisig/`, `~/Documents/Vultisig/`, and platform-specific app data dirs
- FastVault vs SecureVault detection happens during import — the loader must return the correct subclass
