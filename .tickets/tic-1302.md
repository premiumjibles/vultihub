# Chat App: Scaffold standalone app and port agent UI

## Objective

Create a new standalone React application for the Vultisig AI agent chat, extracting the ~10K lines of agent UI from vultisig-windows into its own deployable app. This app is the frontend for the AI agent — it renders conversations, handles user input, and displays transaction status. All crypto logic will come from @vultisig/sdk (wired in the next ticket).

## User Story

Users want a lightweight, web-deployable AI chat interface for their Vultisig wallet that doesn't require the full desktop app. The chat app talks to the agent backend and uses the SDK for wallet operations.

## Design Constraints

- React 19 + Vite (matching vultisig-windows stack)
- styled-components for styling (matches existing agent UI code)
- React Query for data fetching
- Must support web deployment (Vercel/similar) — no Wails dependency
- The agent backend (`agent.vultisig.com`) remains unchanged — this app is a new frontend for it
- i18n: can start English-only, but use react-i18next structure for future locales

## Critical: Unlock Screen & Credential Isolation

The app must have a dedicated unlock screen that collects passwords BEFORE the chat interface loads. This is a security boundary:

1. **Unlock screen** renders two password fields (local key + server key) with optional "same password" toggle
2. On submit, passwords go directly to `sdk.loadVault(path, localPassword)` and `sdk.authenticate(serverPassword)` — in-process, no network except the HTTPS call to `/vault/session`
3. After successful unlock, passwords are discarded from React state immediately
4. The orchestrator receives an already-unlocked vault instance — it has no method to access or request passwords
5. Agent tool schemas have no password fields — the LLM cannot request credentials
6. On session expiry, the unlock screen reappears — credentials are never cached in conversation context, localStorage, or any serializable store

After unlock, show a brief vault summary (name, addresses, balances) so the user confirms they loaded the right vault before entering chat.

## Gotchas

- The orchestrator currently creates tool handlers that import directly from `@core/chain/` and `@core/mpc/` — in this ticket, stub those out or leave them broken. Ticket tic-cedf (rewire to SDK) handles the actual wiring.
- `AgentContextService` requires injected functions (`getVault`, `getVaults`, `getVaultCoins`, `getAddressBookItems`) — these will be wired to SDK in the next ticket, but the interface must be preserved.
- The shared UI components (`Button`, `Stack`, `Text`, `Modal`) are in `lib/ui/` which is an upstream mirror — extract only what's needed, don't copy the entire lib.
- `styled-components` theme must be set up with a `ThemeProvider` — the existing agent code expects `getColor()` and theme tokens to be available.
- The 23 icon components are simple SVGs — straightforward to copy.
- `react-markdown` is used for `MarkdownContent.tsx` — include it as a dependency.
- Password fields must use `type="password"` and `autocomplete="off"` — never render in plaintext or allow browser autofill to leak to other contexts.
